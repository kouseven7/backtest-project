## 4. 調査完了事項

### 4.1 トレンドフィルターの詳細確認（完了）
**確認内容**:
- GCStrategyとVWAPBreakoutStrategyでトレンドフィルターの実装を確認

**証拠1**: GCStrategy Line 67-68
```python
"trend_filter_enabled": False,  # デフォルトは無効（マルチ戦略システムで既にフィルタリング済み）
"allowed_trends": ["uptrend"]  # 許可するトレンド（上昇トレンド）
```

**証拠2**: VWAPBreakoutStrategy
- トレンドフィルターの実装なし（`trend_filter_enabled`パラメータなし）

**結論**: 
- GCStrategyのデフォルトではトレンドフィルターは**無効**
- VWAPBreakoutStrategyはトレンドフィルターを使用していない
- **トレンドフィルターは取引件数減少の原因ではない**

### 4.2 エントリー条件の詳細確認（完了）
**GCStrategy**:
- ゴールデンクロス判定: 短期SMA > 長期SMA
- デフォルトパラメータ: `short_window=5, long_window=25`
- トレンドフィルター: 無効（デフォルト）

**VWAPBreakoutStrategy**:
- VWAPブレイク条件: `current_price > vwap`
- **確認バー条件**: `confirmation_bars=1` - **1本の確認バーが必要**
- 確認バー判定: 過去1本のバーも`price > vwap`であることを要求

**証拠**: VWAPBreakoutStrategy Line 313-322
```python
if self.params.get("confirmation_bars", 0) > 0:
    confirm_ok = True
    for i in range(1, min(self.params["confirmation_bars"] + 1, idx + 1)):
        if self.data[self.price_column].iloc[idx - i] <= vwap:
            logger.debug(f"[entry] idx={idx}: 確認バーNG idx-i={idx-i}")
            confirm_ok = False
            break
    if not confirm_ok:
        return 0
```

**結論**: 
- **VWAPBreakoutStrategyの確認バー条件が厳しい**
- 1本の確認バーが必要なため、エントリーシグナル発生率が低下
- 長期テストログ: `[entry] idx=99: 確認バーNG idx-i=98` - この条件で拒否されている

## 5. 原因の推定（可能性順）

### 可能性1: VWAPBreakoutStrategy確認バー条件が厳しすぎる（最も可能性が高い）
**理由**:
- **証拠あり**: `confirmation_bars=1` - 1本の確認バーが必要
- **証拠あり**: 長期テストログ `[entry] idx=99: 確認バーNG idx-i=98` - この条件で拒否されている
- 確認バー条件: 現在のバーと過去1本のバーの両方が`price > vwap`であることを要求
- この条件により、VWAPブレイク直後のエントリーが拒否される

**改善案**:
- `confirmation_bars=0`に設定して確認バー条件を無効化
- または、確認バー判定ロジックを緩和（VWAP近辺でも許可）

### 可能性2: GCStrategyゴールデンクロス発生率が低い（高い）
**理由**:
- **証拠あり**: 長期テストログ `entry_signals=0` - 33営業日でエントリーシグナルが0件
- ゴールデンクロス（短期SMA > 長期SMA）が発生していない
- デフォルトパラメータ: `short_window=5, long_window=25`
- 2025年1-2月の日経225銘柄の価格動向がゴールデンクロスを発生させなかった可能性

**改善案**:
- SMAパラメータを調整（短期を短く、長期を短くしてクロス頻度を上げる）
- トレンドフィルターを有効化して、上昇トレンド時のみエントリー

### 可能性3: 銘柄切替の頻度が高すぎる（中）
**理由**:
- **証拠あり**: 33日中15回の切替（45.5%）
- 切替時は強制決済される（is_forced_exit=True）
- 切替直後はエントリー機会が減少する可能性

**改善案**:
- 銘柄切替の頻度を下げる（切替条件を厳しくする）
- 切替直後のエントリー待機期間を短縮

### 可能性4: トレンドフィルターが厳しすぎる（低、該当なし）
**理由**:
- **証拠あり**: GCStrategyのデフォルトでは`trend_filter_enabled=False`
- VWAPBreakoutStrategyはトレンドフィルターを使用していない
- **トレンドフィルターは原因ではない**

## 8. 最終結論

### 根本原因（証拠ベース）

**主要原因1**: VWAPBreakoutStrategy確認バー条件が厳しい
- **証拠**: `confirmation_bars=1` - 1本の確認バーが必要
- **証拠**: 長期テストログ `[entry] idx=99: 確認バーNG idx-i=98`
- **影響**: VWAPブレイク直後のエントリーが拒否される

**主要原因2**: GCStrategyゴールデンクロス発生率が低い
- **証拠**: 長期テストログ `entry_signals=0`
- **影響**: 33営業日でエントリーシグナルが0件

**副次的要因**: 銘柄切替の頻度が高い（45.5%）
- **証拠**: 33日中15回の切替
- **影響**: 切替時に強制決済、エントリー機会減少

**除外された要因**: トレンドフィルター
- **証拠**: GCStrategyのデフォルトで`trend_filter_enabled=False`
- **結論**: トレンドフィルターは原因ではない

### 推奨アクション

1. **優先度1（即座に実行可能）**: VWAPBreakoutStrategy確認バー条件を無効化（`confirmation_bars=0`）
2. **優先度2（パラメータ調整）**: GCStrategyのSMAパラメータを調整（クロス頻度を上げる）
3. **優先度3（将来検討）**: 銘柄切替の頻度を最適化

### Phase 2との関係

- Phase 2（データ取得範囲拡大）は既に実装済みで正常動作
- Phase 2は取引件数増加の根本原因ではない
