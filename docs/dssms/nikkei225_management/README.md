## 目標・目的

### 1. 正確な日経225銘柄リストの維持
- **目的**: DSSMSスクリーニングシステムが常に正確な日経225構成銘柄でフィルタリングを実行
- **理由**: 銘柄リストの品質がバックテスト結果に直接影響

### 2. バックテストの再現性確保
- **目的**: 過去の任意の時点における日経225構成銘柄を再現可能にする
- **理由**: 過去のバックテスト結果を検証・比較するため
- **ユーザー要件**: 「もっと古いバックテストも行うので」

### 3. 定期的な銘柄リスト更新の自動化
- **目的**: 年1回の日経225構成銘柄見直しに対応
- **理由**: 手動更新のミスを防ぎ、運用負荷を削減

## バージョン管理要件

### バックテスト再現性の確保

**要件定義**:
- 過去の任意の時点における日経225構成銘柄を再現可能にする
- バージョン管理により、バックテスト実行時点の銘柄セットを特定

**実装方針**:

#### オプション1: 履歴ファイル方式
```
config/dssms/nikkei225_history/
  ├── nikkei225_components_2023-10-01.json  # 2023年10月見直し
  ├── nikkei225_components_2024-10-01.json  # 2024年10月見直し
  └── nikkei225_components_2025-10-01.json  # 2025年10月見直し
```

**メリット**:
- 各時点の銘柄リストが独立ファイルで管理
- バックテスト時に該当ファイルを指定

**デメリット**:
- ファイル数が増加
- 重複データあり

#### オプション2: 単一JSONファイルの履歴管理
```json
{
  "current_version": "2.0.0",
  "versions": [
    {
      "version": "1.0.0",
      "effective_date": "2023-10-01",
      "symbols": [...]
    },
    {
      "version": "2.0.0",
      "effective_date": "2025-12-15",
      "symbols": [...]
    }
  ]
}
```

**メリット**:
- 単一ファイルで管理
- 履歴が一元化

**デメリット**:
- ファイルサイズ増大
- 読み込み処理が複雑化

#### 推奨: オプション1（履歴ファイル方式）

**理由**:
- シンプルで理解しやすい
- 各時点の銘柄リストが独立
- バックテスト実行時の指定が容易
