## 3. 取引回数減少の原因分析

### 3.1 構造的原因: DSSMS切替 = エントリー機会

DSSMSは「単一銘柄集中運用」のため：
1. **常に1銘柄のみ保有**（分散投資なし）
2. **銘柄切替 = 既存ポジション決済 + 新銘柄エントリー**
3. **切替回数 ≈ エントリー機会の総数**

#### メカニズム図解
```
┌─────────────────────────────────────────┐
│ Day 1: 銘柄A保有中                      │
│   ↓ 銘柄Bのシグナル発生                 │
│ Day 2: 切替判定 → min_holding_days未達  │ ← Task 1で厳格化
│   → 切替スキップ                         │
│   → 銘柄Bのエントリー機会喪失           │
└─────────────────────────────────────────┘

Task 1前: min_holding_days緩い → 頻繁に切替 → 多数のエントリー機会
Task 1後: min_holding_days=10 → 切替抑制 → エントリー機会減少
```

### 3.2 前回調査（DSSMS_Low_Entry_Investigation.md）の文脈

**前回調査の主要発見** (Task 1実装前):
- 85回の切替、平均保有期間3.9日（頻繁すぎる）
- GCシグナル35回中、9回のみエントリー（25.7%）
- **72.4%のシグナルを逃す** → 原因: 切替時に既に前銘柄のシグナルが過ぎていた

**前回の推奨**:
> 「min_holding_daysを10日以上に設定して切替頻度を抑制すべき」

**パラドックスの発生**:
- 前回調査: 「切替が多すぎる → シグナルを逃す → エントリー減少」
- Task 1実装: 切替を減らす → 期待: エントリー増加
- **実際の結果**: エントリーも減少 (-66.7%)

### 3.3 パラドックスの解明

前回調査とTask 1実装後で**異なる問題**が顕在化：

#### 前回の問題（切替多すぎ）
```
Day 1: 銘柄4506保有
Day 2: 銘柄5202でGCシグナル発生 → 保有中なので無視
Day 3: 5202に切替 → しかしシグナルは既に終了
Day 4: また別の銘柄に切替（保有期間3.9日）
```
→ **結果**: シグナルの「タイミングずれ」で72.4%逃す

#### Task 1後の問題（切替少なすぎ）
```
Day 1: 銘柄4506保有（1日目）
Day 2-9: 他銘柄でシグナル発生 → min_holding_days未達で切替不可
Day 10: やっと切替可能 → しかし最適銘柄は4506のまま
```
→ **結果**: 切替自体が発生せず、エントリー機会が創出されない

**本質的な問題**: 
DSSMSの「単一銘柄集中」設計では、**切替回数 = エントリー機会総数**となるため、切替を抑制すると必然的にエントリー機会も減少する。

## 5. Task 3への疑問に対する回答

### 5.1 「GC戦略に特化しすぎていないか？」

**確認結果**: 
- Task 1前後の全取引でGCStrategyが選択されている
- 他戦略（VWAP、Breakout、Momentum、Contrarian）は**どちらも選択されず**

**理由**: 
1. DSSMSのランキング・スコアリングでGC戦略が常に優位
2. これは戦略選択ロジックの問題であり、min_holding_daysとは独立

**Task 3のリスク**:
- min_holding_daysをさらに伸ばす（12-15日）と、エントリー機会がさらに減少
- しかし、**他戦略が選択されない現状では、GC戦略の質を高めることは正当**

### 5.2 「min_holding_daysは他戦略にマイナスではないか？」

**重要な構造理解**: 
`min_holding_days`は**SymbolSwitchManagerの切替制約**であり、**各戦略のエントリー/エグジット判断には影響しない**。

#### コード確認結果
- [strategies/gc_strategy_signal.py](strategies/gc_strategy_signal.py#L63): `max_hold_days: 20`（強制決済）
- 他戦略ファイルにも`min_holding_days`パラメータは存在しない
- **各戦略は独自のエントリー/エグジット条件を持つ**

**影響範囲**:
```
min_holding_days=10 → DSSMS切替頻度 ↓
    ↓
全戦略共通: エントリー機会総数 ↓
    ↓
GC戦略: 現在選択されている → 影響あり
他戦略: 現在選択されていない → 実質影響なし（選択されれば同様に影響）
```

### 5.3 「保有期間を固定することではない」への回答

**ユーザーの本質的要求の理解**:
> 「目的は取引で利益をあげること」

**Task 1の評価**: ✅ この要求を満たしている
- 総損益: +11,711円改善
- 勝率: 33.3% → 100%
- 平均利益/取引: +3,714円改善

**min_holding_days=10の意味**:
- ❌ 誤解: 「必ず10日間保有しなければならない」
- ✅ 実際: 「DSSMS切替時に最低10日は同じ銘柄を保持」
- 各戦略は**10日以内でもエグジット可能**（実際: 1日、4日、8日の保有例あり）

## 6. 代替案の検討

### 6.1 現状の制約

DSSMSの「単一銘柄集中」設計における根本的問題:
```
切替頻度 ↑ → シグナルタイミングずれ ↑ → 機会損失 ↑
切替頻度 ↓ → エントリー機会総数 ↓ → 取引回数 ↓
```

### 6.2 代替アプローチ

#### Option A: 複数銘柄ポートフォリオ
```python
# 現在: 単一銘柄のみ保有
holdings = {"4506": 100%}

# 代替案: 上位N銘柄を分散保有
holdings = {
    "4506": 40%,
    "2768": 30%,
    "1662": 30%
}
```
**メリット**: 
- 複数銘柄で同時にシグナルキャッチ可能
- 切替頻度と無関係にエントリー機会増加

**デメリット**: 
- プロジェクト設計思想「分散投資なし」に反する
- 資金管理の複雑化

#### Option B: 最適min_holding_daysの動的調整
```python
# 現在: 固定値
min_holding_days = 10

# 代替案: 市場状況に応じて調整
if market_volatility > threshold:
    min_holding_days = 5  # ボラティリティ高 → 短期保有
else:
    min_holding_days = 15  # ボラティリティ低 → 長期保有
```

#### Option C: Task 3を実行せず現状維持
- 現在のmin_holding_days=10で+10,860円達成
- さらなる最適化（Task 3: 12-15日）はエントリー機会をさらに減らすリスク
